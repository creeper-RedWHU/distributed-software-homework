# 所有问题修复总结

## 问题演进时间线

### 1️⃣ ByteSource 编译错误
**时间**: 初次实现
**错误**: `无法解析符号 'ByteSource'`
**原因**: `org.apache.shiro.util.ByteSource` 导入问题
**解决**: 创建 `CustomCredentialsMatcher` 自定义密码匹配器
**文档**: `COMPILE_FIX.md`

### 2️⃣ securityManager Bean 冲突
**时间**: 第一次启动
**错误**: `The bean 'securityManager' could not be registered`
**原因**: Shiro 自动配置与自定义配置冲突
**解决**: 尝试禁用 `ShiroAutoConfiguration` 和 `ShiroWebAutoConfiguration`
**文档**: `STARTUP_FIX.md`

### 3️⃣ defaultAdvisorAutoProxyCreator Bean 冲突
**时间**: 第二次启动
**错误**: `The bean 'defaultAdvisorAutoProxyCreator' could not be registered`
**原因**: 注解处理器自动配置冲突
**解决**: 尝试禁用 `ShiroAnnotationProcessorAutoConfiguration`
**文档**: `STARTUP_FIX.md`

### 4️⃣ javax.servlet.Filter 找不到（第一次尝试）
**时间**: 第三次启动
**错误**: `java.lang.ClassNotFoundException: javax.servlet.Filter`
**原因**: Spring Boot 4.0.3 使用 Jakarta EE，而 `shiro-spring-boot-starter` 的某些类还在使用 `javax.servlet`
**尝试**: 移除 `shiro-spring-boot-starter`，使用核心 Shiro 库
**结果**: ❌ 仍然失败，核心 Shiro 库也不支持 Jakarta EE

### 5️⃣ Shiro 核心库仍不支持 Jakarta EE ✅ 最终解决
**时间**: 第四次启动
**错误**: `java.lang.NoClassDefFoundError: javax/servlet/Filter` （在 `ShiroFilterFactoryBean` 中）
**根本原因**: **Shiro 2.0.1 的所有库都不支持 Jakarta EE**
**最终解决**: **降级 Spring Boot 到 2.7.18**
**文档**: `SPRING_BOOT_DOWNGRADE.md` ⭐⭐⭐

## 🎯 最终解决方案

### 核心改动

#### 1. pom.xml - 降级 Spring Boot

**修改前**:
```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>4.0.3</version>
    <relativePath/>
</parent>
```

**修改后**:
```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>2.7.18</version>
    <relativePath/>
</parent>
```

#### 2. 修改导入 - jakarta → javax

**Controller 和 DTO 文件**:
```java
// 修改前
import jakarta.servlet.http.HttpServletRequest;
import jakarta.validation.constraints.NotBlank;

// 修改后
import javax.servlet.http.HttpServletRequest;
import javax.validation.constraints.NotBlank;
```

#### 3. Shiro 依赖保持简洁

```xml
<!-- Apache Shiro (核心库) -->
<dependency>
    <groupId>org.apache.shiro</groupId>
    <artifactId>shiro-spring</artifactId>
    <version>2.0.1</version>
</dependency>
<dependency>
    <groupId>org.apache.shiro</groupId>
    <artifactId>shiro-web</artifactId>
    <version>2.0.1</version>
</dependency>
<dependency>
    <groupId>org.apache.shiro</groupId>
    <artifactId>shiro-ehcache</artifactId>
    <version>2.0.1</version>
</dependency>
```

#### 3. 其他文件

| 文件 | 状态 |
|------|------|
| `CustomCredentialsMatcher.java` | ✅ 保留（自定义密码匹配） |
| `UserRealm.java` | ✅ 保留（不使用 ByteSource）|
| `ShiroConfig.java` | ✅ 保留（完整配置）|

## 🎓 经验教训

### 1. Spring Boot Starter 的陷阱
- Starter 提供便利，但也可能带来兼容性问题
- 大版本升级时（如 Jakarta EE 迁移），核心库往往更稳定

### 2. 问题排查思路
```
1. 编译错误 → 检查导入和依赖
2. Bean 冲突 → 禁用自动配置
3. 类找不到 → 检查版本兼容性
4. 多次冲突 → 考虑移除 Starter
```

### 3. 最佳实践
- ✅ 使用核心库，完全控制配置
- ✅ 避免过度依赖自动配置
- ✅ 遇到问题先查版本兼容性
- ✅ 文档记录每个问题的修复过程
