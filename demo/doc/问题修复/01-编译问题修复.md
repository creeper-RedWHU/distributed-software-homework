# 编译问题修复说明

## 问题描述

在初始实现中，`UserRealm.java` 使用了 `org.apache.shiro.util.ByteSource`，但该类在某些 Shiro 版本或配置中可能无法正确导入，导致编译错误。

## 修复方案

采用**自定义 CredentialsMatcher** 的方式，避免使用 `ByteSource`：

### 1. 移除 ByteSource 依赖

**修改文件**: `src/main/java/com/example/demo/shiro/UserRealm.java`

- ❌ 移除：`import org.apache.shiro.util.ByteSource;`
- ❌ 移除：`ByteSource credentialsSalt = ByteSource.Util.bytes(salt);`
- ✅ 简化：直接返回用户名和密码，不再传递盐值

### 2. 创建自定义密码匹配器

**新建文件**: `src/main/java/com/example/demo/shiro/CustomCredentialsMatcher.java`

```java
@Component
public class CustomCredentialsMatcher implements CredentialsMatcher {

    @Autowired
    private UserLoginMapper userLoginMapper;

    @Override
    public boolean doCredentialsMatch(AuthenticationToken token, AuthenticationInfo info) {
        // 获取用户输入的密码
        String inputPassword = new String(((UsernamePasswordToken) token).getPassword());

        // 获取数据库中的密码和盐值
        UserLogin userLogin = userLoginMapper.selectByUsername(...);
        String storedPassword = (String) info.getCredentials();
        String salt = userLogin.getSalt();

        // 使用 PasswordUtil 验证密码
        return PasswordUtil.verifyPassword(inputPassword, storedPassword, salt);
    }
}
```

**优点**：
- ✅ 完全控制密码验证逻辑
- ✅ 不依赖 Shiro 的 ByteSource
- ✅ 可以使用自定义的 PasswordUtil
- ✅ 更加灵活和易于理解

### 3. 修改 Shiro 配置

**修改文件**: `src/main/java/com/example/demo/config/ShiroConfig.java`

**变更内容**：
- ❌ 移除：`HashedCredentialsMatcher` 相关代码
- ✅ 新增：注入 `CustomCredentialsMatcher`
- ✅ 修改：`UserRealm` Bean 使用自定义匹配器

```java
@Bean
public UserRealm userRealm(CustomCredentialsMatcher credentialsMatcher) {
    UserRealm realm = new UserRealm();
    realm.setCredentialsMatcher(credentialsMatcher);  // 使用自定义匹配器
    // ... 其他配置
    return realm;
}
```

### 4. 完善 Result 类

**修改文件**: `src/main/java/com/example/demo/common/Result.java`

新增方法重载：
```java
public static <T> Result<T> fail(ErrorCode errorCode, String message) {
    return new Result<>(errorCode.getCode(), message, null, System.currentTimeMillis());
}
```

## 技术说明

### Shiro 密码验证流程

#### 修复前（使用 ByteSource）
```
1. UserRealm.doGetAuthenticationInfo()
   └─ 返回 SimpleAuthenticationInfo(username, password, salt, realmName)
2. Shiro 内置 HashedCredentialsMatcher
   └─ 使用 ByteSource 封装的盐值
   └─ 自动进行 SHA-256 + Salt 验证
```

#### 修复后（自定义匹配器）
```
1. UserRealm.doGetAuthenticationInfo()
   └─ 返回 SimpleAuthenticationInfo(username, password, realmName)
2. CustomCredentialsMatcher.doCredentialsMatch()
   └─ 从数据库查询盐值
   └─ 调用 PasswordUtil.verifyPassword()
   └─ 返回验证结果
```

## 常见问题

### Q1: 为什么不直接使用 Shiro 的 HashedCredentialsMatcher？

**答**:
- HashedCredentialsMatcher 需要使用 ByteSource 传递盐值
- ByteSource 在不同 Shiro 版本中可能存在兼容性问题
- 自定义匹配器更加灵活，便于扩展

### Q2: 自定义匹配器会影响安全性吗？

**答**:
- ❌ 不会。安全性取决于加密算法（SHA-256 + Salt）
- ✅ 自定义匹配器只是调用方式不同
- ✅ 使用相同的 PasswordUtil 加密和验证

### Q3: 如果要切换加密算法怎么办？

**答**:
- 只需修改 `PasswordUtil` 中的加密实现
- `CustomCredentialsMatcher` 会自动使用新算法
- 不需要修改 Shiro 配置
