## 2. 秒杀核心流程

### 2.1 秒杀主流程（同步+异步混合）

```plantuml
@startuml seckill-main-flow
!theme plain
|用户|
start
:点击"立即秒杀";

|Controller|
:接收秒杀请求;
:参数校验（activityId, userId）;

|Service - 同步阶段|
:【校验1】活动状态检查;
note right: 从 Redis 缓存读取\n活动时间、状态
if (活动未开始或已结束?) then (是)
    :返回"活动不可用";
    stop
endif

:【校验2】用户限购检查;
note right: Redis: seckill:bought:{activityId}:{userId}
if (已达限购上限?) then (是)
    :返回"已达限购上限";
    stop
endif

:【核心】Redis Lua 原子扣减库存;
note right
  **Lua脚本原子操作：**
  1. 检查库存 > 0
  2. 检查用户未超限
  3. 库存 -1
  4. 用户购买数 +1
  全部在一个原子操作中完成
end note
if (扣减成功?) then (否)
    :返回"库存不足";
    stop
else (是)
endif

:生成订单号（雪花算法）;
:发送消息到 Kafka;
note right
  Topic: seckill-orders
  消息体: {orderNo, userId,
    activityId, quantity}
end note

:返回"秒杀成功，正在创建订单";

|Kafka Consumer - 异步阶段|
:消费 seckill-orders 消息;

:MySQL 乐观锁扣减库存;
note right: 作为 Redis 的兜底保障\nWHERE available_stock >= quantity\nAND version = #{version}
if (数据库扣减成功?) then (是)
    :创建订单记录;
    :记录库存流水;
    :发布 OrderCreatedEvent;
else (否)
    :Redis 库存回滚;
    :发布 SeckillFailedEvent;
endif

|Event Bus|
:事件监听器处理;
:WebSocket 推送结果给用户;

|用户|
:收到秒杀结果通知;
if (秒杀成功?) then (是)
    :跳转支付页面;
else (否)
    :显示秒杀失败;
endif

stop
@enduml
```

### 2.2 Redis Lua 原子扣减脚本

```lua
-- seckill_deduct.lua
-- KEYS[1] = seckill:stock:{activityId}
-- KEYS[2] = seckill:bought:{activityId}:{userId}
-- ARGV[1] = 限购数量
-- ARGV[2] = 购买数量

-- 检查库存
local stock = tonumber(redis.call('GET', KEYS[1]) or '0')
if stock < tonumber(ARGV[2]) then
    return -1  -- 库存不足
end

-- 检查用户已购数量
local bought = tonumber(redis.call('GET', KEYS[2]) or '0')
if bought + tonumber(ARGV[2]) > tonumber(ARGV[1]) then
    return -2  -- 超出限购
end

-- 原子扣减
redis.call('DECRBY', KEYS[1], ARGV[2])
redis.call('INCRBY', KEYS[2], ARGV[2])

return 1  -- 扣减成功
```

## 4. 库存管理

### 4.1 库存预热（活动开始前）

```plantuml
@startuml stock-warmup
!theme plain

|定时任务|
start
:扫描即将开始的秒杀活动;
note right: start_time在未来5分钟内

while (有待预热活动?) is (是)
    |StockService|
    :从MySQL读取库存;
    :写入Redis;
    note right
      SET seckill:stock:{id} = available_stock
      HSET seckill:activity:{id} 活动详情
    end note
    :发布 StockWarmedUpEvent;
    :Hook: onStockWarmup;
    note right: 插件可在此执行自定义预热逻辑\n如: 预加载商品详情到缓存
endwhile (否)

stop
@enduml
```

### 4.2 双写一致性保障

库存存在于 Redis 和 MySQL 两个存储中，一致性策略：

```
          Redis (快速路径)          MySQL (持久化)
              │                        │
   秒杀请求 ──┤                        │
              │ Lua原子扣减            │
              │ ←──成功──→             │
              │                        │
              │    Kafka 异步消息       │
              │ ──────────────────→    │
              │                    乐观锁扣减
              │                    ←成功/失败→
              │                        │
              │   失败则Redis回滚       │
              │ ←──────────────────    │
```

**一致性保障手段**：
1. Redis 扣减成功是前提
2. MySQL 乐观锁扣减作为最终确认
3. 失败时 Redis 回滚（补偿机制）
4. 定时对账任务兜底（Redis 库存 vs MySQL 库存）

## 6. 用户购买流程（完整）

```plantuml
@startuml user-purchase-flow
!theme plain
actor 用户
participant "前端页面" as FE
participant "SeckillController" as SC
participant "SeckillService" as SS
participant "Redis" as Redis
participant "Kafka" as Kafka
participant "OrderConsumer" as OC
participant "MySQL" as DB
participant "WebSocket" as WS

== 秒杀前 ==
用户 -> FE : 打开秒杀页面
FE -> SC : GET /api/seckill/activities
SC -> Redis : 查询活动缓存
Redis --> SC : 活动列表
SC --> FE : 返回活动信息
FE -> WS : 建立WebSocket连接
note over WS : 用于接收实时库存\n和秒杀结果

== 倒计时阶段 ==
loop 每秒推送
    WS -> FE : 推送倒计时/库存数
end

== 秒杀开始 ==
用户 -> FE : 点击"立即秒杀"
FE -> SC : POST /api/seckill/execute
SC -> SS : execute(activityId, userId)
SS -> Redis : 执行Lua扣减脚本

alt 扣减成功
    Redis --> SS : return 1
    SS -> Kafka : 发送下单消息
    SS --> SC : 返回排队中
    SC --> FE : "秒杀成功，正在处理"

    == 异步下单 ==
    Kafka -> OC : 消费下单消息
    OC -> DB : 乐观锁扣减库存
    OC -> DB : 创建订单
    OC -> WS : 推送下单成功

    WS -> FE : "订单已创建，请支付"
    FE -> 用户 : 跳转支付页面

else 库存不足
    Redis --> SS : return -1
    SS --> SC : 返回库存不足
    SC --> FE : "抱歉，已售罄"
end

@enduml
```

## 7. 非功能性需求

| 需求 | 指标 | 实现手段 |
|------|------|----------|
| 响应时间 | 秒杀接口 < 50ms | Redis 内存操作 + 异步下单 |
| 吞吐量 | > 10,000 QPS | 分层过滤 + Kafka 削峰 |
| 可用性 | 99.9% | 多实例部署 + 中间件集群 |
| 一致性 | 零超卖 | Redis Lua 原子 + MySQL 乐观锁 |
| 数据完整性 | 订单不丢失 | Kafka 持久化 + MySQL 事务 |
