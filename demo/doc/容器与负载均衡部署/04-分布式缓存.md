# 分布式缓存 (Redis)

## 一、Redis 缓存架构

```
Client ──► Nginx ──► Backend ──┬──► Redis (缓存)
                               └──► MySQL (数据库)
```

请求流程：
1. 客户端请求 `/api/product/{id}`
2. Nginx 将请求代理到后端
3. 后端先查 Redis 缓存
4. 缓存命中 → 直接返回
5. 缓存未命中 → 查询MySQL → 写入Redis → 返回

## 二、商品详情页缓存实现

### 2.1 核心代码

位于 `ProductService.java`：

```java
public Result<ProductVO> getProductById(Long id) {
    String cacheKey = "product:detail:" + id;

    // 1. 查询缓存
    Object cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return Result.success(convertCachedToVO(cached));
    }

    // 2. 缓存未命中，查询数据库
    Product product = productMapper.selectById(id);

    // 3. 写入缓存（带TTL）
    redisTemplate.opsForValue().set(cacheKey, vo, ttl, TimeUnit.SECONDS);

    return Result.success(vo);
}
```

### 2.2 缓存键设计

| 键格式 | 用途 | 示例 |
|--------|------|------|
| `product:detail:{id}` | 商品详情缓存 | `product:detail:1` |
| `product:null:{id}` | 空值缓存（防穿透） | `product:null:99999` |
| `product:lock:{id}` | 互斥锁（防击穿） | `product:lock:1` |

| `seckill:order:{sid}:{uid}` | 秒杀订单标记 | `seckill:order:1:100` |

## 三、缓存穿透

### 3.1 问题描述

查询一个**数据库中不存在**的数据，每次请求都会穿透缓存直接访问数据库。

```
Client ──► Redis (MISS) ──► MySQL (NULL) ──► 无数据可缓存
              ↑                                     │
              └─────── 每次都穿透到DB ◄─────────────┘
```

恶意攻击者可以利用此漏洞用大量不存在的ID进行请求，压垮数据库。

### 3.2 解决方案：缓存空值

```java
Product product = productMapper.selectById(id);
if (product == null) {
    // 缓存空值，TTL设为5分钟（较短）
    redisTemplate.opsForValue().set(nullKey, "NULL", 300, TimeUnit.SECONDS);
    return Result.fail("商品不存在");
}
```

**原理**：即使数据库查不到，也在Redis中缓存一个空标记。后续相同请求直接从Redis返回，不再访问数据库。

**空值TTL设置较短（5分钟）** 的原因：
- 避免占用过多Redis内存
- 如果数据后来被创建，能在较短时间内生效

### 3.3 验证穿透防护

```bash
# 第一次请求不存在的商品（会查DB）
curl http://localhost/api/product/99999
# 返回: {"code":400,"message":"商品不存在"}

# 第二次请求（直接走Redis空值缓存，不查DB）
curl http://localhost/api/product/99999
# 返回相同结果，但后端日志不会有DB查询记录
```

## 四、缓存击穿

### 4.1 问题描述

某个**热点key**过期的瞬间，大量并发请求同时涌入，全部穿透到数据库。

```
               缓存过期瞬间
Thread-1 ──► Redis (MISS) ──┐
Thread-2 ──► Redis (MISS) ──┤
Thread-3 ──► Redis (MISS) ──┼──► MySQL（并发压力巨大）
...                         │
Thread-N ──► Redis (MISS) ──┘
```

### 4.2 解决方案：互斥锁

```java
// 使用Redis SETNX实现互斥锁
String lockKey = "product:lock:" + id;
boolean locked = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);

if (!locked) {
    // 未获取到锁，短暂等待后重试
    Thread.sleep(50);
    return getProductById(id);  // 递归重试
}

try {
    // 双重检查：再次查缓存（可能被其他线程重建了）
    cached = redisTemplate.opsForValue().get(cacheKey);
    if (cached != null) {
        return Result.success(convertCachedToVO(cached));
    }

    // 查DB并写缓存
    Product product = productMapper.selectById(id);
    redisTemplate.opsForValue().set(cacheKey, vo, ttl, TimeUnit.SECONDS);
} finally {
    redisTemplate.delete(lockKey);  // 释放锁
}
```

**原理**：
1. 只有获取到锁的线程才能查DB并重建缓存
2. 其他线程等待后重试，大概率能从缓存获取数据
3. 使用双重检查避免重复查询
4. 锁设置10秒过期，防止死锁

### 4.3 流程图

```
Thread-1 ──► Redis MISS ──► 获取锁成功 ──► 查DB ──► 写缓存 ──► 释放锁
Thread-2 ──► Redis MISS ──► 获取锁失败 ──► 等待50ms ──► Redis HIT ──► 返回
Thread-3 ──► Redis MISS ──► 获取锁失败 ──► 等待50ms ──► Redis HIT ──► 返回
```

## 五、缓存雪崩

### 5.1 问题描述

大量缓存**同时过期**（例如都设置了相同的TTL），导致瞬间所有请求都穿透到数据库。

```
Time T: 所有缓存同时过期
         │
         ▼
所有请求 ──► Redis (全部MISS) ──► MySQL（被压垮）
```

### 5.2 解决方案：随机TTL偏移

```java
// 基础TTL = 1小时，加上 0~600秒 的随机偏移
long ttl = 3600 + ThreadLocalRandom.current().nextLong(0, 600);
redisTemplate.opsForValue().set(cacheKey, vo, ttl, TimeUnit.SECONDS);
```

**原理**：每个key的过期时间不同，避免同时过期。

- 商品1的TTL = 3600 + 123 = 3723秒
- 商品2的TTL = 3600 + 456 = 4056秒
- 商品3的TTL = 3600 + 89 = 3689秒

过期时间被打散，不会集中失效。

### 5.3 其他防护措施

| 措施 | 说明 |
|------|------|
| 随机TTL | 避免同时过期（已实现） |
| 缓存预热 | 系统启动时提前加载热点数据 |
| 多级缓存 | 本地缓存(Caffeine) + Redis，即使Redis雪崩本地缓存仍可用 |
| 限流降级 | 通过Nginx或Sentinel限制请求速率 |

## 六、Redis 配置说明

### application.yml

```yaml
spring:
  redis:
    host: localhost       # Docker环境中为 redis
    port: 6379
    database: 0
    timeout: 3000ms
    lettuce:
      pool:
        max-active: 20    # 最大连接数
        max-idle: 10      # 最大空闲连接
        min-idle: 5       # 最小空闲连接
        max-wait: 3000ms  # 最大等待时间
```

### RedisConfig.java

配置 `RedisTemplate<String, Object>`，使用JSON序列化：

```java
@Bean
public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
    RedisTemplate<String, Object> template = new RedisTemplate<>();
    template.setKeySerializer(new StringRedisSerializer());
    template.setValueSerializer(new Jackson2JsonRedisSerializer<>(Object.class));
    return template;
}
```
