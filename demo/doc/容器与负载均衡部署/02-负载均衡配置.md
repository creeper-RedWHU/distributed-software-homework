# 负载均衡配置

## 一、Nginx 负载均衡原理

Nginx作为反向代理服务器，接收客户端请求后，根据配置的负载均衡算法将请求转发到后端服务器集群。

```
Client ──► Nginx(:80) ──┬──► App1(:8081)
                        └──► App2(:8082)
```

## 二、upstream 配置

在 `docker/nginx/nginx-lb.conf` 中配置后端服务器集群：

```nginx
upstream seckill_backend {
    server app1:8081;
    server app2:8082;
    keepalive 32;
}
```

## 三、四种负载均衡算法

### 3.1 轮询（Round Robin）—— 默认算法

```nginx
upstream seckill_backend {
    server app1:8081;
    server app2:8082;
    keepalive 32;
}
```

**特点**：
- 请求按顺序依次分配到每个后端
- 每个后端被访问的概率相等
- 适合后端服务器性能相近的场景

**预期结果**：100次请求中，app1和app2各处理约50次。

### 3.2 加权轮询（Weighted Round Robin）

```nginx
upstream seckill_backend {
    server app1:8081 weight=3;
    server app2:8082 weight=1;
    keepalive 32;
}
```

**特点**：
- weight值越大，被分配到的请求越多
- 适合后端服务器性能不均的场景（高配置服务器分配更多请求）

**预期结果**：100次请求中，app1处理约75次，app2处理约25次。

### 3.3 最少连接（Least Connections）

```nginx
upstream seckill_backend {
    least_conn;
    server app1:8081;
    server app2:8082;
    keepalive 32;
}
```

**特点**：
- 将请求分配给当前活跃连接数最少的后端
- 适合处理时间差异较大的请求

**预期结果**：每个后端的并发连接数基本相等。

### 3.4 IP Hash

```nginx
upstream seckill_backend {
    ip_hash;
    server app1:8081;
    server app2:8082;
    keepalive 32;
}
```

**特点**：
- 根据客户端IP的hash值决定分配到哪个后端
- 同一个客户端IP总是被分配到同一个后端
- 适合需要会话保持的场景

**预期结果**：同一客户端的所有请求都发到同一后端。

## 四、切换算法的方法

编辑 `docker/nginx/nginx-lb.conf`，注释/取消注释对应的 `upstream` 块，然后重新加载 Nginx：

```bash
# 修改配置后重新加载
docker exec seckill-nginx nginx -s reload

# 或重启nginx容器
docker-compose -f docker-compose.lb.yml restart nginx
```

## 五、验证负载均衡

### 5.1 通过响应头验证

每个后端实例会在响应头中添加 `X-Server-Port`：

```bash
# 连续请求10次，观察端口变化
for i in $(seq 1 10); do
  echo -n "请求$i: "
  curl -s -o /dev/null -D - http://localhost/api/product/server-info 2>&1 | grep X-Server-Port
done
```

轮询算法下输出：
```
请求1: X-Server-Port: 8081
请求2: X-Server-Port: 8082
请求3: X-Server-Port: 8081
请求4: X-Server-Port: 8082
...
```

### 5.2 通过后端日志验证

```bash
# 查看app1处理的请求数
docker logs seckill-app1 2>&1 | grep "查询商品" | wc -l

# 查看app2处理的请求数
docker logs seckill-app2 2>&1 | grep "查询商品" | wc -l
```

### 5.3 通过 Nginx 日志验证

```bash
# Nginx access log 包含 upstream 信息
docker exec seckill-nginx cat /var/log/nginx/lb_access.log | tail -20
```

日志中 `upstream=` 字段显示每个请求转发到了哪个后端。

### 5.4 通过前端页面验证

访问 `http://localhost`，点击"连续请求10次"按钮，页面会显示每次请求的分发统计。

## 六、Nginx 关键配置说明

### 反向代理头部传递

```nginx
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
proxy_set_header X-Forwarded-Proto $scheme;
```

- `X-Real-IP`：客户端真实IP
- `X-Forwarded-For`：代理链路
- `X-Forwarded-Proto`：原始协议（http/https）

### 连接保持

```nginx
proxy_http_version 1.1;
proxy_set_header Connection "";
keepalive 32;
```

使用 HTTP/1.1 长连接减少连接建立开销。
