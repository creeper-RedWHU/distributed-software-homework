# 动静分离配置

## 一、什么是动静分离

动静分离是将网站的**静态资源**（HTML、CSS、JS、图片等）和**动态请求**（API接口）分开处理的策略：

- **静态资源**：由 Nginx 直接提供，不经过后端
- **动态请求**：由 Nginx 代理转发到后端应用

```
                     ┌─── /css/style.css ───► Nginx 本地文件
Client ──► Nginx ────┤
                     └─── /api/product/1 ───► 后端集群
```

## 二、目录结构

```
docker/
└── static/                    # 静态资源目录
    ├── index.html             # 首页
    ├── css/
    │   └── style.css          # 样式表
    └── js/
        └── app.js             # JavaScript脚本
```

在 docker-compose.lb.yml 中挂载到 Nginx 容器：

```yaml
nginx:
  volumes:
    - ./docker/static:/usr/share/nginx/html/static:ro
```

## 三、Nginx 配置

在 `docker/nginx/conf.d/lb.conf` 中配置：

```nginx
server {
    listen 80;

    # ========== 静态资源 - Nginx直接提供 ==========

    # 首页
    location / {
        root /usr/share/nginx/html/static;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 静态文件 (CSS/JS/图片等)
    location ~* \.(html|css|js|jpg|jpeg|png|gif|ico|svg)$ {
        root /usr/share/nginx/html/static;
        expires 7d;                              # 浏览器缓存7天
        add_header Cache-Control "public, immutable";
        add_header X-Static-Cache "HIT";         # 标记来源
        access_log off;                          # 关闭日志减少IO
    }

    # ========== 动态请求 - 代理到后端 ==========

    location /api/ {
        proxy_pass http://seckill_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        add_header X-Upstream-Addr $upstream_addr;
    }
}
```

### 关键配置说明

| 配置项 | 说明 |
|--------|------|
| `root` | 静态文件根目录 |
| `try_files` | 尝试匹配文件，找不到则返回 index.html（支持SPA） |
| `expires 7d` | 设置 HTTP Cache-Control 头，浏览器缓存7天 |
| `access_log off` | 静态资源不记录访问日志，减少IO开销 |
| `X-Static-Cache: HIT` | 自定义响应头，标识该资源由Nginx直接提供 |

## 四、验证动静分离

### 4.1 验证静态资源

```bash
# 请求CSS文件，检查响应头
curl -I http://localhost/css/style.css
```

预期输出：
```
HTTP/1.1 200 OK
Content-Type: text/css
Cache-Control: public, immutable
Expires: ...（7天后的日期）
X-Static-Cache: HIT
```

关键点：
- `X-Static-Cache: HIT` 表示由 Nginx 直接返回
- 没有 `X-Server-Port` 头（因为请求没有经过后端）

### 4.2 验证动态请求

```bash
# 请求API接口，检查响应头
curl -I http://localhost/api/product/list
```

预期输出：
```
HTTP/1.1 200 OK
Content-Type: application/json
X-Server-Port: 8081
X-Upstream-Addr: app1:8081
```

关键点：
- `X-Server-Port` 表示请求经过了后端处理
- `X-Upstream-Addr` 显示具体转发到了哪个后端

## 五、前端页面说明

### index.html
秒杀系统前端页面，包含：
- 负载均衡状态检测（连续请求观察分发）
- 商品列表展示
- 秒杀专区
- Redis缓存操作演示（查询/清除/预热）

### style.css
页面样式，包含：
- 响应式布局（支持移动端）
- 商品卡片、按钮等组件样式
- Gzip压缩友好（纯文本）

### app.js
前端交互逻辑：
- `checkServer()` - 检查请求分发到哪个后端
- `batchCheck()` - 连续10次请求统计分发情况
- `loadProducts()` - 加载商品列表
- `loadSeckillList()` - 加载秒杀列表
- `queryProduct()` - 查询商品详情（演示Redis缓存）
- `doSeckill()` - 执行秒杀

## 六、性能对比

使用JMeter压测静态资源和动态API，对比响应时间：

| 请求类型 | 预期平均响应时间 | 说明 |
|---------|--------------|------|
| 静态HTML | 1-5ms | Nginx直接返回，无需经过后端 |
| 静态CSS | 1-3ms | 文件较小，返回更快 |
| 静态JS | 1-3ms | 文件较小，返回更快 |
| 动态API (缓存命中) | 5-20ms | 经过Nginx代理 + 后端Redis查询 |
| 动态API (缓存未命中) | 20-50ms | 经过Nginx代理 + 后端DB查询 |

**结论**：静态资源由Nginx直接返回，响应时间远优于动态API。动静分离能有效降低后端压力，提升整体性能。
