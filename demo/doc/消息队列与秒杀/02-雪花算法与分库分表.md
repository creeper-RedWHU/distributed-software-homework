# 雪花算法订单ID + ShardingSphere 分库分表

## 一、架构概述

```
                         ┌─────────────────────────────────┐
                         │       ShardingSphere-JDBC        │
                         │  逻辑表: t_seckill_order         │
                         └───────────┬─────────────────────┘
                                     │ 分表路由: id % 2
                          ┌──────────┴──────────┐
                          ▼                     ▼
                 ┌─────────────────┐   ┌─────────────────┐
                 │ t_seckill_order_0│   │ t_seckill_order_1│
                 │   id % 2 = 0    │   │   id % 2 = 1    │
                 └─────────────────┘   └─────────────────┘
                          │                     │
                          └──────────┬──────────┘
                                     ▼
                         ┌─────────────────────┐
                         │  RoutingDataSource   │
                         │  (读写分离路由)       │
                         └───┬─────────────┬───┘
                             │             │
                    ┌────────▼──┐    ┌─────▼───────┐
                    │  Master   │    │   Slave     │
                    │  (读写)   │───►│   (只读)    │
                    └───────────┘复制 └─────────────┘
```

**分层设计:**
- **ShardingSphere层**: 将逻辑表 `t_seckill_order` 路由到物理表 `t_seckill_order_0/1`
- **RoutingDataSource层**: 将读操作路由到 Slave，写操作路由到 Master
- 两层可独立工作，互不干扰

## 二、雪花算法 (Snowflake)

### 2.1 ID结构

```
  1位    41位时间戳    5位数据中心   5位机器ID    12位序列号
  ┌─┬──────────────┬───────────┬───────────┬────────────┐
  │0│ timestamp-ms  │datacenter │  worker   │  sequence  │
  └─┴──────────────┴───────────┴───────────┴────────────┘
  63                22          17          12           0
```

- **时间戳**: 支持约69年
- **数据中心+机器**: 最多1024个节点
- **序列号**: 每毫秒最多4096个ID
- **理论峰值**: 409.6万 ID/秒

### 2.2 基因法（Gene Method）

普通雪花ID的问题：按 userId 查订单时，不知道订单在哪个分片。

**基因法解决方案**: 将 userId 的哈希嵌入订单ID的低位，使得:
- 同一用户的订单ID取模结果相同
- 按订单ID分表 = 按用户ID分表
- 查询时无需扫描全部分片

```java
public long nextOrderId(long userId) {
    long id = nextId();
    long userHash = userId & 0xF; // 取userId低4位
    return (id & ~0xFL) | userHash; // 替换订单ID低4位
}
```

**效果验证:**
```
userId=1  → orderId=...0001, 路由到 t_seckill_order_1 (奇数)
userId=2  → orderId=...0010, 路由到 t_seckill_order_0 (偶数)
userId=3  → orderId=...0011, 路由到 t_seckill_order_1 (奇数)
userId=4  → orderId=...0100, 路由到 t_seckill_order_0 (偶数)
```

同一userId的所有订单都路由到同一分片，查询时只需扫描一个表。

### 2.3 核心代码

```java
@Component
public class SnowflakeIdGenerator {
    private static final long EPOCH = 1700000000000L;
    // ... 位运算配置

    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("Clock moved backwards");
        }
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & MAX_SEQUENCE;
            if (sequence == 0) timestamp = waitNextMillis(lastTimestamp);
        } else {
            sequence = 0L;
        }
        lastTimestamp = timestamp;
        return ((timestamp - EPOCH) << TIMESTAMP_SHIFT)
                | (datacenterId << DATACENTER_ID_SHIFT)
                | (workerId << WORKER_ID_SHIFT)
                | sequence;
    }
}
```

## 三、ShardingSphere 分库分表

### 3.1 分表策略

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 逻辑表 | `t_seckill_order` | 代码中使用的表名 |
| 物理表 | `t_seckill_order_0`, `t_seckill_order_1` | 实际存储的表 |
| 分表键 | `id`（订单ID） | 雪花算法+基因法生成 |
| 分表算法 | `MOD` (取模) | `id % 2` 决定路由到哪个表 |
| 分库键 | `user_id` | 生产环境扩展为 `user_id % N` 路由到不同库 |

### 3.2 ShardingSphere 配置代码

```java
// DataSourceConfig.java 中 ShardingSphere 配置
@Bean @Primary
public DataSource dataSource(DataSource routingDataSource) throws SQLException {
    Map<String, DataSource> dataSourceMap = new HashMap<>();
    dataSourceMap.put("ds0", routingDataSource); // 包装读写分离数据源

    ShardingRuleConfiguration shardingRule = new ShardingRuleConfiguration();

    // 订单表分表规则
    ShardingTableRuleConfiguration orderRule = new ShardingTableRuleConfiguration(
            "t_seckill_order", "ds0.t_seckill_order_${0..1}");
    orderRule.setTableShardingStrategy(
            new StandardShardingStrategyConfiguration("id", "order-id-mod"));
    shardingRule.getTables().add(orderRule);

    // MOD分表算法
    Properties modProps = new Properties();
    modProps.setProperty("sharding-count", "2");
    shardingRule.getShardingAlgorithms().put("order-id-mod",
            new AlgorithmConfiguration("MOD", modProps));

    return ShardingSphereDataSourceFactory.createDataSource(
            dataSourceMap, Collections.singletonList(shardingRule), props);
}
```

### 3.4 分库策略说明（生产扩展）

当前演示环境为单库模式。生产环境扩展为多库时：

```
用户请求
   │
   ▼ user_id % 2
   ├── db_0 (user_id为偶数)
   │   ├── t_seckill_order_0
   │   └── t_seckill_order_1
   └── db_1 (user_id为奇数)
       ├── t_seckill_order_0
       └── t_seckill_order_1
```

扩展方式：修改 ShardingSphere 配置，增加数据源和分库策略：
```java
dataSourceMap.put("ds0", masterDb0);
dataSourceMap.put("ds1", masterDb1);

orderRule = new ShardingTableRuleConfiguration(
    "t_seckill_order", "ds${0..1}.t_seckill_order_${0..1}");
orderRule.setDatabaseShardingStrategy(
    new StandardShardingStrategyConfiguration("user_id", "user-id-mod"));
```

## 六、技术要点总结

| 技术 | 解决的问题 |
|------|-----------|
| 雪花算法 | 分布式环境下生成全局唯一、趋势递增的订单ID |
| 基因法 | 将userId信息嵌入订单ID，避免跨分片查询 |
| ShardingSphere-JDBC | 透明分表，应用代码无需感知物理表 |
| MOD分表算法 | 数据均匀分布到各分片 |
| 与读写分离共存 | ShardingSphere包装RoutingDataSource，两层独立工作 |
